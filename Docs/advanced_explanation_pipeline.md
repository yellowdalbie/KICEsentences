# 궁극의 사용자 경험을 위한 3 페이즈(Phase) 해설 생성 파이프라인 기획서

**버전:** 2.0
**상태:** 진행 중 (대화 반영 및 연구 단계)
**목표:** 단순 텍스트 표출을 넘어, 완벽한 사용자 경험(UX)과 벡터 DB 매칭을 동시에 충족하는 "수학적 논리 패턴 매칭 및 해설 파이프라인" 설계

---

## 🏗️ 전체 아키텍처 개요 (3-Phase Core-Expansion Model)

이 2.0 파이프라인은 인간의 문제 풀이 순서가 아닌, **엔진의 데이터 무결성**을 최우선으로 하여 파이프라인을 3개의 페이즈로 분리합니다. 변하지 않는 수학적 본질(Core)을 먼저 확정하고, 개별 문항의 맥락(Context)을 나중에 붙여 환각(Hallucination)을 원천 차단합니다.

### 🔴 Phase A. Core Resolution (본질 추출 및 뼈대 구축)
기존의 엉터리 해설 스텝들을 분해하고 수학교육학적 본질에 맞추어 단일 성취기준 및 개념 사전에 강제 매핑합니다.
*   **[Step 1] 성취기준 매핑 (Ontological Anchoring):** 쪼개진 깨끗한 스텝에 `concepts.json`의 교육과정 성취기준 ID를 1:1로 고정.
*   **[Step 2] 수학적 개념 추출 및 표준 사전화:** 기존 `Sol/` 폴더를 마이닝하여 "표준 개념 딕셔너리" 생성 및 해당 스텝에 매핑.
*(※ 1단계와 2단계는 나선형 부트스트래핑 방식으로 교차 검증하며 고도화)*

### 🔵 Phase B. Contextual Expansion (맥락 변수 역추적)
Phase A에서 고정된 개념(Action)을 근거로 삼아, 원본 문제에서 원인과 결과를 찾아(역추적) 데이터를 구체화합니다.
*   **[Step 0] 맥락 트리거(Trigger) 정형화:** 이 수학적 개념을 '왜' 썼는지에 대한 원본 문제의 텍스트/수식적 단서(원인) 찾기.
*   **[Step 3] 논리적 징검다리(Bridge) 및 Delta 구체화:** 이 수학적 개념을 써서 '무엇'을 얻었는지 중간 결과물(결과) 식별하기.

### 🟢 Phase C. Final Rendering (UI 표출)
*   **[Step 4] 타이틀 렌더링 및 검증:** 확보된 `[Trigger - Concept - Result]` 트립렛 JSON 데이터를 문법 템플릿(`[~수단]을 이용하여 [~대상]을 [~결과]하기`)에 조립 후 송출.

## 📉 Phase A 토큰 최적화 하이브리드 로직 (비용 절감 및 속도 극대화)

수천 개의 파일을 처리할 때 막대한 LLM 요금과 지연 시간이 발생하는 것을 막기 위해, Phase A는 아래 4단계의 '저비용-고효율' 로직을 순차적으로 탑재합니다.

1.  **[저비용] 가벼운 모델 기반 순수 텍스트 분할 (Flash Model)**
    *   사전 배제. 오직 텍스트 논리적 마디 자르기에만 싸고 빠른 모델 활용.
2.  **[제로 코스트] 로컬 Regex/Keyword 기반 도메인 필터링**
    *   Python 스크립트가 텍스트 키워드를 분석해 전체 600개 성취기준 중 3~5개 후보군만 추림.
3.  **[제로 코스트] 로컬 벡터 임베딩 코사인 매핑 (Vector Search)**
    *   기존 구축된 임베딩을 활용해 후보군과 유사도 검사. 90% 이상 시 LLM 없이 즉각 확정.
4.  **[고비용/최소화] 예외 상황(Fallback)에만 Heavy LLM 투입**
    *   벡터 유사도가 낮아 판별이 애매한 극소수(Hard Example)의 스텝만 고성능 LLM(Pro 모델)에 보내서 최종 판결.
